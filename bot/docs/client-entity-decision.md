# Решение по модели клиентов

## Контекст

`telegram_users` использовалась как таблица конечных клиентов, которые записываются на услуги через Telegram.

## Решение

Использовать **единую сущность `clients`** как главный источник правды по клиенту.

Практически это означает:

1. ✅ Выполнено: таблица переименована в `clients` (минимальный риск и быстрый выигрыш в ясности).
2. На втором шаге (когда появятся другие каналы) — вынести канальные идентификаторы в отдельную таблицу
   `client_contacts`/`client_identities` (`telegram_id`, `phone`, `email`, `whatsapp_id` и т.д.).

## Почему так (DRY/KISS/SOLID)

- **KISS**: простое переименование уже убирает терминологическую путаницу без сложной миграции.
- **DRY**: единая таблица клиентов исключает дубли разных «типов пользователей».
- **SOLID (SRP)**: `clients` отвечает за клиента, а канальные идентификаторы (при необходимости) — за связь с каналами.

## Правила дедупликации

Для одного `account_id`:

- искать существующего клиента по нормализованным `phone` и `email`;
- если найден — обновлять существующую запись, а не создавать новую;
- если не найден — создавать нового клиента;
- `telegram_id` считать сильным идентификатором канала, но не единственным глобальным ключом клиента.

## Индексы и ограничения

Рекомендуемые частичные уникальные индексы (PostgreSQL):

- `UNIQUE(account_id, telegram_id) WHERE telegram_id IS NOT NULL`
- `UNIQUE(account_id, phone) WHERE phone IS NOT NULL`
- `UNIQUE(account_id, lower(email)) WHERE email IS NOT NULL`

Это позволит безопасно связывать записи из разных каналов с одним клиентом.
