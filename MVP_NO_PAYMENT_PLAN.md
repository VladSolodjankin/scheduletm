# MVP plan: запуск бота без оплаты

Цель: довести текущий backend Telegram-бот до состояния, когда можно принимать реальные записи без платежей и без веб-кабинета.

## Приоритет 1 — «бот принимает и сохраняет записи стабильно»

1. ✅ Привести время к `app_settings.timezone` (IANA) и убрать фиксированный UTC+3.
2. ✅ Добавить защиту от двойного бронирования слота (DB exclusion constraint по пересечению интервалов).
3. ✅ Добавить отмену записи пользователем (`status = cancelled`).
4. ✅ Добавить базовую команду сброса сессии (`/start`, `/reset`) и кнопку «Назад» хотя бы на 1 шаг.

Критерий готовности:
- ✅ пользователь может записаться, посмотреть запись, перенести и отменить;
- ✅ два пользователя не могут занять один и тот же слот;
-  ✅время в интерфейсе и БД предсказуемо для выбранной timezone.

## Приоритет 2 — «минимальная операционка»

1. Добавить напоминания в Telegram за 24 часа (канал только Telegram).
2. Включить управляемый `AUTO_SET_WEBHOOK` (в dev не трогать webhook).
3. ✅ Добавить `npm run typecheck` и запускать его в локальном CI/перед деплоем.

## Что можно отложить (не блокирует MVP без оплаты)

- BPMN-движок и BPMN-редактор.
- Мульти-аккаунт веб-кабинет и публичная регистрация пользователей.
- Email/SMS-уведомления.
- Полноценная ролевая модель и админка.

## Рекомендуемый первый технический шаг (сегодня)

Статус: ✅ выполнено.

Сделанный «вертикальный» инкремент:
1. ✅ timezone на IANA + корректные слоты,
2. ✅ блокировка двойной записи,
3. ✅ отмена записи.

Результат: рабочее ядро сервиса записи без оплаты и без web UI уже собрано; `/start` и `/reset` сбрасывают сессию, а в выборе специалиста есть кнопка «Назад» на шаг услуг.

## Следующие шаги до production (без сценария оплаты)

1. **Надёжность и эксплуатация**
   - Добавить идемпотентность обработки webhook-update (защита от повторной обработки `update_id`).
   - Ввести structured logging + корреляцию (`request_id`, `update_id`, `account_id`, `user_id`).
   - Подготовить health/readiness checks для оркестратора и алерты на ошибки webhook.

2. **Уведомления и повторные попытки**
   - ✅ Доведен `notifications` до базового рабочего контура: планирование, ретраи, статусы (`pending/retry/sent/failed/cancelled`).
   - ✅ Реализованы напоминания T-24h по каналам Telegram/email/SMS (email/SMS как stub-провайдеры на этапе MVP).
   - Реализовать возможность записи сразу на 2+ сессий, количество сессий должно храниться в services
   - Добавить в даблицу specialists базовую стоимость за 1 сессию/ 1 час и сделать расчет стомости услуг в services в зависимости от стоимости сессии/часа у специалиста.

3. **UX и устойчивость сценария**
   - Добавить «Назад» и «Отмена» на остальные шаги (дата, время, контактные шаги).
   - Добавить ограничение «один активный booking flow на пользователя» с явным восстановлением состояния после рестартов.
   - Улучшить тексты ошибок (невалидный контакт, устаревшая сессия, нет слотов).

4. **Управляемость конфигурации**
   - Сделать `SLOT_STEP_MIN` полностью настраиваемым через `app_settings`.
   - Добавить исключения календаря: выходные/праздники/персональные окна специалистов.
   - Включить флаг `AUTO_SET_WEBHOOK` для безопасного dev/prod поведения.

5. **Качество и безопасность**
   - Расширить тесты webhook-сценариев (happy path + edge cases).
   - Добавить rate limit и базовую защиту от спама в webhook.
   - Подготовить runbook: бэкапы БД, откат миграций, чек-лист релиза.
