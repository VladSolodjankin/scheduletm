# Service Setup Spec (MVP)

## Цель
Сделать настройку услуг простой для малого бизнеса (1 специалист) и гибкой для команд (много специалистов), без переусложнения UX.

---

## 1) Данные и связи
Используем **3 таблицы**:

- `specialists`
- `services`
- `specialist_services` (связка many-to-many)

Почему 3 таблицы:
- одна услуга может быть у нескольких специалистов;
- один специалист может оказывать несколько услуг;
- у специалиста может быть индивидуальная цена/длительность для конкретной услуги.

### `services` (базовый уровень)
Минимальные поля услуги:
- `name`
- `description`
- `base_price`
- `base_duration_minutes`
- `first_session_free` (boolean)
- `image_url` (или media reference)
- `account_id`
- `is_active`

### `specialist_services` (уровень назначения)
Минимальные поля назначения:
- `service_id`
- `specialist_id`
- `price_override` (nullable)
- `duration_override_minutes` (nullable, опционально)
- `is_active`

### Итоговая цена
- если `specialist_services.price_override` заполнена — использовать её;
- иначе использовать `services.base_price`.

---

## 2) Роли и доступ

### Owner
- просмотр/редактирование услуг и назначений для всех `account_id`.

### Admin
- просмотр/редактирование только в рамках своего `account_id`.

### Specialist
- просмотр только своих назначенных услуг;
- редактирование только своих параметров назначения (например, `price_override`),
  без изменения глобальных полей услуги (`name`, `description`, `base_price` и т.д.).

---

## 3) UX создания услуги (рекомендуемый гибрид)
Поддержать **оба сценария**:

1. **Быстрый:** создать услугу и сразу назначить специалистов.
2. **Детальный:** после создания открыть экран “Специалисты услуги” и точечно настроить каждого.

### Форма “Создать услугу”
Поля:
- Название
- Описание
- Базовая стоимость
- Длительность
- Первая консультация/сессия бесплатна
- Картинка

Блок специалистов:
- `multiselect` со специалистами;
- поиск по специалистам;
- кнопка **“Назначить всем активным”**;
- кнопка **“Очистить выбор”**;
- счетчик вида: “Назначено X из Y”.

---

## 4) Edge cases (обязательные правила)

### Сценарий: 1 активный специалист
- назначать его к новой услуге автоматически;
- выбор специалистов можно упростить или показывать как read-only.

### Сценарий: 0 активных специалистов
- разрешить сохранить услугу как черновик/неактивную;
- показать предупреждение: услуга недоступна для записи, пока не назначен специалист;
- дать CTA “Добавить специалиста”.

### Сценарий: admin = specialist
- поддерживать как валидный сценарий (часто для small business/solo).

### Сценарий: удаление единственного специалиста
- запрещать удаление/деактивацию последнего активного специалиста
  (минимум при наличии активных услуг).

---

## 5) Поведение кнопки “Назначить всем”
- распространяется на **всех активных специалистов текущего `account_id`**;
- не должна затирать индивидуальные настройки (`price_override`) без отдельного подтверждения;
- после действия показывать подтверждение (toast) с количеством назначенных специалистов.

---

## 6) Клиентский сценарий записи
Порядок выбора:
1. Услуга
2. Специалист
3. День
4. Время

Цена на шаге выбора специалиста:
- `price_override`, если есть;
- иначе `base_price` услуги.

---

## 7) Анти-переусложнение (KISS)
- по умолчанию показывать простой сценарий;
- расширенные настройки (индивидуальная цена/длительность) — на отдельном экране услуги;
- массовые операции добавить только базовые:
  - назначить всем,
  - снять всех,
  - очистить override-ы (опционально позже).

---

## 8) Краткие acceptance criteria (MVP)
- Можно создать услугу с базовыми полями.
- Можно назначить услугу нескольким специалистам.
- Можно назначить всем активным специалистам одной кнопкой.
- Для специалиста можно задать индивидуальную цену услуги.
- Если индивидуальная цена не задана, используется базовая.
- При 1 специалисте назначение происходит автоматически.
- При 0 специалистах услуга сохраняется, но недоступна для онлайн-записи.
- Нельзя удалить/деактивировать последнего активного специалиста (по бизнес-правилу).
