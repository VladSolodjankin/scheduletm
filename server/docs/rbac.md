# RBAC (roles & permissions)

Текущая модель прав в server модуле централизована в `src/policies/rolePermissions.ts`.

## Роли

- `owner` — уникальный суперпользователь; может управлять системой глобально.
- `admin` — администратор своего пространства (`account_id`).
- `specialist` — работает со своими записями и клиентами своего `account_id`.
- `client` — работает только со своими записями и personal user settings.

## Матрица прав

| Действие                                                             | owner | admin | specialist | client |
|----------------------------------------------------------------------|------:|------:|-----------:|-------:|
| System settings                                                      | ✅ | ❌ | ❌ | ❌ |
| User settings                                                        | ✅ | ✅ | ✅ | ✅ |
| Управление специалистами                                             | ✅ | ✅ | ❌ | ❌ |
| Управление всеми пользователями (по account_id)                      | ✅ | ✅ | ❌ | ❌ |
| Управление всеми пользователями (вне зависимости от account_id)      | ✅ | ❌ | ❌ | ❌ |
| Управление клиентами                                                 | ✅ | ✅ | ✅ | ❌ |
| Создание `admin`/`specialist`                                        | ✅ | ✅ | ❌ | ❌ |
| Создание `client`                                                    | ✅ | ✅ | ✅ | ❌ |
| Просмотр/изменение всех appointments (по account_id)                 | ✅ | ✅ | ❌ | ❌ |
| Просмотр/изменение всех appointments (вне зависимости от account_id) | ✅ | ❌ | ❌ | ❌ |
| Работа только со своими appointments специалиста                     | n/a | n/a | ✅ | ❌ |
| Работа только со своими appointments клиента                         | n/a | n/a | ❌ | ✅ |
| Создание appointments                                                | ✅ | ✅ | ✅ (для клиентов) | ✅ (для себя) |
| Mark-paid/notify                                                     | ✅ | ✅ | ✅ (для своих appointments) | ❌ |

## Принцип реализации

- Простая явная логика (KISS): checks в сервисах + централизованные policy-функции.
- Без преждевременной сложности: таблицы `roles/permissions` не нужны, пока роли фиксированы.
- Переход на динамические права имеет смысл только при реальной бизнес-потребности (кастомные роли и UI-редактор прав).
